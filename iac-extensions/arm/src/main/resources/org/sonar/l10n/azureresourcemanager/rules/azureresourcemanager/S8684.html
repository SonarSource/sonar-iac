Control actions such as If, Switch, Foreach, Until, and Scope are building blocks of Logic Apps workflows. When they are nested deeply inside one another the workflow becomes hard to read, maintain, debug, and test.

Why is this an issue?

Deeply nested control actions create "spaghetti workflows" where reviewers must mentally track many levels of branching and looping. Small changes at deep levels have unpredictable effects, and run history tools flatten the action tree making failures hard to trace.

The recommended remediation is to extract deeply nested logic into a child Logic App or to flatten the structure by rethinking the control flow.

Noncompliant Code Example

With the default threshold of 3:

{
  "type": "Microsoft.Logic/workflows",
  "apiVersion": "2019-05-01",
  "name": "deeply-nested",
  "properties": {
    "definition": {
      "actions": {
        "Check": {
          "type": "If",
          "actions": {
            "Loop": {
              "type": "Foreach",
              "actions": {
                "Inner_Check": {
                  "type": "If",
                  "actions": {
                    "Retry": {
                      "type": "Until",
                      "actions": {}
                    }
                  }
                }
              }
            }
          }
        }
      },
      "triggers": {}
    }
  }
}

Compliant Solution

{
  "type": "Microsoft.Logic/workflows",
  "apiVersion": "2019-05-01",
  "name": "flat-with-child",
  "properties": {
    "definition": {
      "actions": {
        "Check": {
          "type": "If",
          "actions": {
            "Loop": {
              "type": "Foreach",
              "actions": {
                "Call_Child": {
                  "type": "Workflow",
                  "inputs": {}
                }
              }
            }
          }
        }
      },
      "triggers": {}
    }
  }
}
